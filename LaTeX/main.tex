\section{Introduction}

The aim of this laboratory exercise was to design and implement a frequency synthesiser using an ADuC841 microcontroller programmed in Assembly. The lab was intended to build understanding of timing and frequency generation in embedded systems, starting with software-based methods and progressing to hardware-based solutions using timers and interrupts. 

As an initial step, we used the example program \texttt{blink.asm} as a reference. This program demonstrated basic Assembly, input/output pin control and the use of long loops to create delays in software.

In the first part of the assignment, a square-wave was generated using software delays only, without the use of hardware timers or interrupts. The output frequency was controlled by calculating the instruction execution times and adjusting the delay loops accordingly.

In the second part of the assignment, a square wave was generated using hardware timers which generated interrupts at regular intervals. First an example program was studied and run, then our own program was written which accommodated adjustable frequencies and several user-focused features.

Relying on dedicated timer hardware rather than software delays has the benefit of allowing the CPU core to perform other tasks, as well as providing a more precise frequency, as software delays can be unpredictable for reasons discussed below.

\section{Task 1: Software Timing}

\subsection{Program Design}

A sample program \texttt{blink.asm} as provided for initial demonstration of using software delays to generate a frequency. The target frequency was much lower and used to drive an LED rather than a transducer, but the software is similar.

This program was modified to provide a frequency of 900 Hz using a nested 'for' loop to delay the CPU core. The precise number of clock cycles to generate a 900 Hz square wave was calculated and the delay function tweaked accordingly. Only two levels of 'for' loop nesting was necessary for this faster period, rather than three levels in the original \texttt{blink.asm}.

The main program loop has only three instructions:\begin{itemize}
    \item Call a 'delay' to stall the CPU for a time
    \item Complement (flip) the output bit labelled 'SOUND' which is tied to the transducer on the physical board
    \item Return to the start of the loop
\end{itemize}

The ADuC841 runs with a clock speed of 11.0592 MHz. To accomplish a 900 Hz wave, the precise duration of this loop had to be precisely 6144 clock cycles, as given by equation \ref{eq:sw-timing-1}. Note the output bit must flip twice per square wave period.

\begin{equation}
    \label{eq:sw-timing-1}
    11.0592 \times 10^6 \frac{\text{clock cycles}}{\text{s}}=6144 \frac{\text{clock cycles}}{\text{bit complement}}\times (2\times900) \frac{\text{bit complements}}{\text{s}}
\end{equation}

A delay function was designed to generate this precise period, accounting for the overhead in each loop. A table calculating the precise duration of the loop is shown in table \ref{tab:sw-timing-loop-duration}. Two \texttt{NOP} operations were included to artificially increase the loop duration.


\begin{table}
    \centering
    \begin{tabular}{lllllll}
        & Instruction & Clock Cycles & Occurences per loop\\
        \hline
        MAIN: & & &\\
        LOOP: & CALL DELAY    & 3+delay & 1\\
        &CPL SOUND            & 2 & 1\\
        &JMP LOOP             & 3 & 1\\
        \hline
        DELAY: &MOV R5, \#16  & 2 & 1\\
        DLY1: & MOV R7, \#126 & 2 & 16\\
              &DJNZ R7, \$    & 3 & $2016 = 16\times 126$\\
              &DJNZ R5, DLY1  & 3 & 16\\
              &NOP            & 1 & 1\\
              &NOP            & 1 & 1\\
              &RET            & 4 & 1\\
        \hline
        \textbf{Total} && 6144
    \end{tabular}
    \caption{Calculation of main loop duration in clock cycles. See the actual Assembly in listing \ref{lst:software-timing}.}
    \label{tab:sw-timing-loop-duration}
\end{table}

\textbf{Limitations of this design:} Generally, software-based designs are not to be considered reliable for precisely timed applications for several reasons:
\begin{itemize}
    \item In an embedded program which responds to interrupts, the interrupts could cause unpredictable delay between software loops, causing frequency jitter
    \item Software development is more difficult and timing will need to be recalculated whenever a functional change is made
    \item Some instructions on some MCUs have nondeterministic latency so software cannot be precise
    \item Keeping the CPU constantly busy could be considered a waste of power
\end{itemize}

    \item Note the "states" count for the simulation, indicating how many clock cycles have run so far

\section{Example Hardware Program}
An example program was provided to demonstrate the use of hardware timing. Full code is omitted for readability, but amounts to:\begin{itemize}
    \item An ISR is defined for overflows of timer 0\begin{itemize}
        \item The ISR causes the output pin "SOUND" to complement
    \end{itemize}
    \item The main program sets timer 0 to ungated timer mode and enables the interrupt
    \item There is then an indefinite "NOP" loop which is regularly interrupted by the timer
\end{itemize}

This approach relies on a dedicated hardware timer to cause the interrupts, meaning unpredictable delays in software cannot cause the output waveform to jitter.

\subsection{Expected Frequency}
Timer 0 is configured as a 13-bit counter with no pre-scaler. This will overflow every $2^{13}=8192$ clock cycles. At a frequency of 11.0592 MHz, this gives an interrupt every 740.74 microseconds. Each interrupt complements the output, so the period corresponds to two such interrupts, or 1.48148 milliseconds. This gives a frequency of precisely \textbf{675 Hz}.

\subsection{Program Design}

\subsubsection{Requirements}
This task was the most substantial, having the following requirements:
\begin{enumerate}
    \item Generate a square wave at an adjustable frequency using a hardware timer.
    \item Allow the user to select the one of 8 frequencies using a set of 3 switches.
    \item A panel of 8 LED's should indicate the selected frequency in one-hot code.
    \item Another LED should flash at a rate visible to humans to give an indication the program is running (a "heartbeat"). Exact timing is not important.
\end{enumerate}

Additionally, an optional feature was also implemented to have a pushbutton on the board toggle the heartbeat LED on an off. The physical button generates some bouncing when pressed, and no debouncing hardware is provide, so a software solution was developed.

\subsubsection{Design Decisions}

\textbf{Choice of Frequency Synthesis Mechanism}

In order to provide more precise frequency synthesis, this design relied on hardware interrupt rather than software delays or polling to generate the appropriate bit flips at the output.

Three philosophies for generating the adjustable frequency were proposed which variously used a timer generating interrupts with a fixed frequency, and software responding to those interrupts to control a divided frequency, or a timer which simply provides an adjustable frequency.

Timer 2 on the ADuC841 provides adjustable frequency via the autoreload mechanism. To investigate the viability of generating precise frequencies with this mechanism, the range and granularity of frequencies was studied. The frequencies which can be generated are given by:

\begin{equation}
    f=\frac{11.0592 \text{ MHz}}{2\times(2^{16}-n)} \quad \forall \quad n \in \{ 0,...,65535 \}
    \label{eq:hw-frequency-1}
\end{equation}

This allows frequencies in the range 84.375 Hz to 5.5296 MHz, with significantly finer granularity at the lower end. However, only frequencies in the range 600 Hz to 9 kHz were of interest for this project. At the top end of this range, consecutive possible frequencies differ by 0.16\%. Consecutive musical notes in the standard temperament differ by $2^\frac{1}{12}=1+\sim 5.9\%$, meaning this quantisation error should be inaudible to a human. Using overflow interrupts from timer 2 with an autoreload mechanism is therefore suitable for synthesising the frequencies of interest.

\textbf{Choice of Frequencies}

Frequencies of an E major triad were selected ranging from E5 (659.3 Hz) up to E8 (5.274 kHz). Appropriate reload values were calculated for each one by solving equation \ref{eq:hw-frequency-1}. These values were hardcoded into a lookup table in the software, allowing them to be reprogrammed easily.

A mechanism for calculating frequencies used in Western music by multiplying by a fixed values of $2^\frac{1}{12}$ but it was decided the software complications in this arithmetic were not justified.

\textbf{Heartbeat LED Mechanism}

While the timing for the heartbeat LED could have been implemented in software, it was preferred to avoid software delay functions and so another timer and interrupt was used. Timer 1 was configured in 16-bit timer mode to generate interrupts at the maximum period of $\frac{2^{16}}{1.0592 \text{MHz}}=\sim 5.9 \text{ ms}$. When this interrupt was generated, it was used to increment register R1. When register R1 overflowed after 256 such interrupts, the LED would complement. This gave an LED flashing period of 3 seconds (1.5 seconds on, 1.5 seconds off). After physical testing this it was judged to be unnaturally slow, so the ISR was modified to increment R1 by 4, giving a period of 0.76 seconds.

Dedicating a second timer for the simple task of generating a flashing LED could be considered overkill if resources are constrained and the MCU has other tasks it also needs to accomplish. It is also possible to rely on the output of timer 2 and dividing with two registers appropriately to generate a sufficiently slow LED, with the consequence that the LED's blink rate will vary with the frequency being synthesised. If the time interval counter (TIC) is being used for other purposes, it is possible for the software to regularly poll the TIC and rely on sufficient time passing to blink the LED. For this project timer 1 was used as there was no constraint not to do so.

\textbf{Switchable Frequencies Mechanism}

The panel of switches did not generate any interrupt when changed, so it was necessary to rely on software polling to respond to the switch positions. Assuming the main loop is fast enough to be imperceptible to humans, this is not an issue.

At each loop, the state of the 8 switches was read, and put through a bitwise AND with 0x07 to ignore all but the final three bits.

A lookup table was then used to determine the appropriate reload value for timer 2 to generate the appropriate frequency. The lookup table data was included into the program memory using the \texttt{DB} assembler directive, which allows the developer to explicitly include bytes program memory. The data pointer \texttt{DPTR} would be pointed at the head of this lookup table, and the command \texttt{MOVC A,@A+DPTR} used to read the item at an \textit{index equal} to the accumulator \textit{into} the accumulator. This was repeated for the low byte and the high byte and (described later) the one-hot code for the display. A relevant code excerpt is shown in listing \ref{lst:lookup-table-example}.

\begin{lstlisting}[label=lst:lookup-table-example,caption=Minimal example of a lookup table in Assembly]
MOV A, SWITCHES		; Load the switch info into A
ANL A, #07h			; Mask away all but the last three bits
MOV DPTR, #RELOAD_VALUE_TABLE_L
; Prepare the data pointer with the table of lower bytes for the reload values
MOVC A, @A+DPTR		; Read from the table into A
MOV RCAP2L, A		; Write A to the timer 2 config register (lower byte)

RELOAD_VALUE_TABLE_L:
    DB	03Ch,	0FFh,	022h,	09Eh,	011h,	0CFh,	088h,	0E8h;

\end{lstlisting}

A possible improvement to this main loop would be to detect changes in the switch state and skip the frequency lookup if there has been no change, which avoids an unnecessary write to the timer 2 autoreload register that could have unintended hardware consequence. In practice this does not seem to be an issue.

\textbf{Display Panel Mechanism}

A display panel of LED's had to show the frequency being generated. An active-low one-hot code had to be generated from the state of the switches. For example, switch 3 should generate the byte 0x11110111 to be written to the LED display.

Originally a bit-shifting mechanism was considered but this was found to require several clock cycles and a 'for' loop. Instead, a simple lookup table was used, which is looked up in the main software loop at the same time as the frequency lookup occurs.

\textbf{Heartbeat LED Disable Button}

A pushbutton was expected to disable the heartbeat LED mechanism when pressed, or re-enable when pressed again. The physical button experienced bouncing, so debouncing was necessary.

Initially debouncing was to be implemented by comparing the timestamp of an interrupt with the timestamp of the previous successful interrupt, and ignoring rapidly successive interrupts. However, no appropriate timestamp was available as the TIC was not in use in the MCU. If the TIC were configured for other purposes regardless, its use would be appropriate here to avoid relying on software delays.

Instead debouncing was accomplished in software using software delays with the following method:
\begin{itemize}
    \item The initial button press (edge-triggered) would create an interrupt request
    \item The ISR would respond immediately, disabling the heartbeat
    \item The ISR would delay for $\sim$50 ms for any further interrupts to arrive
    \item Any further pushbutton interrupts which arrived in that time were cleared before exiting the ISR
\end{itemize}

The ADuC841 does not allow an ISR to interrupt itself with another interrupt request, so it is sufficient to clear the interrupt flag at the end of one ISR to ignore any more that took place.

The software delay was implemented using a similar 'for' loop to that in task 1.

To disable the heartbeat LED, timer 1 would be disabled, and the LED explicitly turned off immediately. Switching off immediately, rather than waiting for the next natural off, allows the user to observe immediately that the button press was recognised. Similarly, when switching the heartbeat LED on, the LED is switched on immediately, and R1 is reset to ensure it does not switch off to soon, giving the user assurance that the button press worked. These improvements could be removed if program memory was constrained.

\textbf{Program Summary}

The initial setup to accommodate the program described above had the following steps:
\begin{itemize}
    \item Configure T2CON to put timer 2 in timer mode with autoreload (done as a single full-register transaction).
    \item Configure TCON and TMOD to put timer 1 in 16-bit timer mode and disable timer 0, as well as ensuring INT0 (the pushbutton interrupt) is an edge-sensitive interrupt.
    \item Configure IE and IP to recognise the appropriate interrupts.
\end{itemize}

Three interrupt sources were allowed:
\begin{itemize}
    \item Timer 2 overflow, used for frequency synthesis
    \item Timer 1 overflow, used for the heartbeat LED
    \item External interrupt INT0, used for the pushbutton
\end{itemize}

Timer 2 overflow was given \textbf{priority} to ensure the frequency did not jitter if another interrupt was already taking place when timer 2 overflowed.

This setup stage then fell through to a main loop which iteratively:
\begin{itemize}
    \item Read the state of the switches
    \item Looked up the appropriate reload values for that frequency (low byte and high byte) and wrote to RCAP2L and RCAP2H
    \item Looked up the one-hot code for the LED display and wrote to "DISPLAY" (P0)
\end{itemize}
